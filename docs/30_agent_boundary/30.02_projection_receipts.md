# Projection Receipts

Receipts close the loop between human intent, Agent projection, OpenTofu state,
runtime Discord actions, and Wendao knowledge context.

## Projection Phases

| Phase | Output | Review owner |
|---|---|---|
| Interpret | Knowledge operation or intent manifest | Human or Agent |
| Enrich | Wendao context refs, summaries, tags | Agent |
| Project | HCL variables or runtime action plan | Agent/Discord adapter |
| Review | OpenTofu plan or runtime proposal | Human for durable/risky changes |
| Execute | `tofu apply` or bot/runtime call | Tooling |
| Record | Receipt manifest | Agent/tooling |

## Receipt Shape

```toml
kind = "projection_receipt"
intent_kind = "paper_announcement"
status = "applied"
created_at = "2026-05-08T00:00:00Z"

[source]
operation_path = "examples/intents/knowledge_operation.publish_paper_with_discussion.toml"
action_id = "announce_paper"
workflow_node_ref = "bpmn:ServiceTask.AnnouncePaper"
graph_node_ref = "petgraph:announce_paper"
intent_path = "examples/intents/paper_announcement.toml"

[projection]

module_ref = "git:<commit-sha>"

[discord]

channel_id = "000000000000000000"

[verification]
plan_reviewed = true
apply_command = "direnv exec . tofu -chdir=opentofu/discord/examples/knowledge_space_basic apply"
```

## Boundary Rule

Receipts may point to Wendao source refs, but this repository should not depend
on Wendao internals. A receipt is a portable fact about a projection that
another system can ingest later.

When execution starts from a `knowledge_operation`, each receipt should include
the operation path, local action ID, BPMN node ref, and petgraph node ref it
closes. This lets Wendao reconstruct workflow state without parsing provider
state.

## Runtime Actions

When an operation is too dynamic for HCL state, such as a discussion thread or
reply, the Agent runtime may execute it directly through a bot or webhook. The
receipt still records:

- the source intent;
- the Discord target;
- the runtime action;
- the produced Discord object IDs;
- whether a human reviewed the action first.

The current runtime command is:

```shell
direnv exec . just create-discussion-post
```

It reads `examples/intents/discussion_post.toml`, creates or reuses an active
thread/post under the target channel, sends the initial prompt when a new thread
is created, and writes a receipt under `.run/receipts`.

Agenda and kanban use the same receipt boundary:

```shell
direnv exec . just sync-forum-tags
direnv exec . just create-agenda-item
direnv exec . just create-scheduled-event
direnv exec . just create-kanban-card
direnv exec . just create-review-gate
```

They read `agenda_item`, `kanban_card`, or `review_gate` manifests, create or
reuse a forum post/card under the target forum, optionally create a Discord
scheduled event for the agenda session, copy BPMN and petgraph refs into
receipts, and record requested/applied/missing Discord forum tags.
`sync-forum-tags` prepares the profile-declared tags first. Those tags are
display state only; Wendao remains the canonical workflow state.
