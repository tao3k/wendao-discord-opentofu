# Knowledge Operation Projection Contract

`knowledge_operation` is the Discord projection envelope for one Wendao
workflow step or user request. It is not a workflow engine. Wendao BPMN owns the
business process, Wendao's graph layer can own the executable topology, and this
repository owns the Discord-facing projection into OpenTofu, runtime bot/API
calls, and receipts.

## Position

The operation contract sits below Wendao workflow and above Discord execution:

```text
human request
  -> Wendao BPMN workflow
  -> Wendao petgraph execution topology
  -> knowledge_operation projection envelope
  -> OpenTofu plan or Discord runtime call
  -> Discord objects
  -> receipts back to Wendao workflow state
```

The contract is intentionally compact. TOML is the machine-readable source, and
a Markdown preview is the human and Agent review surface. The preview is the
default Agent-facing surface; JSON-heavy forms should remain optional tooling
outputs, not the primary collaboration format.

## Boundary Rule

This repository must not reimplement Wendao workflow orchestration. It may:

- accept workflow refs and graph-node refs from Wendao;
- project Discord-specific actions into HCL or runtime commands;
- verify Discord/OpenTofu results;
- emit receipts that Wendao can ingest back into workflow state.

It must not become the owner of BPMN transitions, global scheduling, retry
policy, or cross-system compensation logic.

## Operation Shape

```toml
kind = "knowledge_operation"
goal = "publish_paper_with_discussion"
title = "Publish and discuss Attention Is All You Need"
requested_by = "human"
review_mode = "plan_then_apply"
receipt_required = true

[workflow]
engine = "wendao"
model = "bpmn"
process_ref = "wendao://workflow/publish-paper-with-discussion"
run_ref = "wendao://workflow-run/example"
graph_ref = "wendao://workflow-graph/example"
node_ref = "bpmn:SubProcess.PublishPaperWithDiscussion"

[context]
server_ref = "discord://server/wendao"
source_refs = ["wendao://paper/attention-is-all-you-need"]

[[actions]]
id = "announce_paper"
kind = "paper_announcement"
mode = "opentofu"
target = "papers"
workflow_node_ref = "bpmn:ServiceTask.AnnouncePaper"
graph_node_ref = "petgraph:announce_paper"

[[actions]]
id = "open_discussion"
kind = "discussion_post"
mode = "runtime"
target = "paper-discussion-forum"
workflow_node_ref = "bpmn:ServiceTask.OpenPaperDiscussion"
graph_node_ref = "petgraph:open_discussion"
depends_on = ["announce_paper"]

[[actions]]
id = "record_receipt"
kind = "projection_receipt"
mode = "record"
workflow_node_ref = "bpmn:ServiceTask.RecordDiscordReceipt"
graph_node_ref = "petgraph:record_receipt"
depends_on = ["announce_paper", "open_discussion"]
```

## Fields

| Field | Meaning |
|---|---|
| `kind` | Always `knowledge_operation`. |
| `goal` | Stable projection family, such as `publish_paper_with_discussion`. |
| `title` | Short human-readable operation title. |
| `requested_by` | Origin of the request: `human`, `agent`, `workflow`, or `system`. |
| `review_mode` | Review posture before Discord-side execution. |
| `receipt_required` | Whether Discord-side execution must emit receipts. |
| `workflow` | Wendao workflow, BPMN, run, graph, and node references. |
| `context` | Server refs, Wendao refs, evidence, and source identifiers. |
| `actions` | Discord projection actions local to this envelope. |

Action IDs are local to the projection envelope. They give the Discord adapter a
small dependency view without claiming ownership of Wendao's full workflow
graph.

## Workflow Refs

| Field | Meaning |
|---|---|
| `workflow.engine` | Upstream workflow authority, normally `wendao`. |
| `workflow.model` | Upstream model, normally `bpmn`. |
| `workflow.process_ref` | Stable process definition reference. |
| `workflow.run_ref` | Optional concrete workflow-run reference. |
| `workflow.graph_ref` | Optional petgraph/topology reference. |
| `workflow.node_ref` | BPMN node or subprocess that produced this projection. |
| `actions.workflow_node_ref` | BPMN node for this projected Discord action. |
| `actions.graph_node_ref` | petgraph node for this projected Discord action. |

Refs are opaque to this repository. They are copied into receipts so Wendao can
join Discord-side facts back to its workflow state.

## Action Modes

| Mode | Owner | Use |
|---|---|---|
| `opentofu` | HCL library | Durable structure and managed announcements. |
| `runtime` | Bot/API runtime | Forum posts, threads, replies, and short-lived interactions. |
| `record` | Agent/tooling | Receipts and provenance records. |
| `proposal` | Agent | Space-management plans before human approval. |

## Review Modes

| Review mode | Meaning |
|---|---|
| `preview_only` | Render an operation without executing it. |
| `plan_then_apply` | Show a projection plan, then execute after approval or policy. |
| `proposal_first` | Produce a management proposal before changing Discord. |
| `human_approval_required` | Never execute until an explicit human approval is present. |

Destructive actions and permission changes should default to
`human_approval_required`. Runtime discussion posts can use `plan_then_apply`
when the upstream workflow policy allows it and receipts are required.

## Preview Surface

Agents should show a compact Markdown preview before execution:

```markdown
# Publish and discuss Attention Is All You Need

Goal: `publish_paper_with_discussion`
Workflow: `wendao` / `bpmn`
Process: `wendao://workflow/publish-paper-with-discussion`

Sources:
- `wendao://paper/attention-is-all-you-need`

Review: `plan_then_apply`

Actions:
- `announce_paper` -> `papers` via opentofu (`bpmn:ServiceTask.AnnouncePaper`)
- `open_discussion` -> `paper-discussion-forum` via runtime after `announce_paper` (`bpmn:ServiceTask.OpenPaperDiscussion`)
- `record_receipt` -> `projection_receipt` via record after `announce_paper`, `open_discussion` (`bpmn:ServiceTask.RecordDiscordReceipt`)

Receipts required: yes
```

This preview is deliberately smaller than the TOML. It helps a human approve
the Discord projection and helps an Agent stay oriented without ingesting a
large JSON object.

## Projection Rules

- `opentofu` actions must resolve to module inputs and an OpenTofu plan.
- `runtime` actions must resolve to one explicit script or bot command.
- `record` actions must capture produced Discord IDs and verification evidence.
- action dependencies are local guardrails for Discord projection order; the
  upstream Wendao graph remains authoritative for global scheduling.
- receipts must include the operation path, workflow refs, and action IDs they
  close.

## Receipt Join Rule

Receipts emitted from a `knowledge_operation` must carry enough workflow refs
for Wendao to join the Discord fact back to BPMN/petgraph state:

```toml
[source]
operation_path = "examples/intents/knowledge_operation.publish_paper_with_discussion.toml"
action_id = "open_discussion"
workflow_node_ref = "bpmn:ServiceTask.OpenPaperDiscussion"
graph_node_ref = "petgraph:open_discussion"
source_refs = ["wendao://paper/attention-is-all-you-need"]
```

The Discord receipt remains portable. Wendao can ingest it, but this repository
does not import Wendao internals.

## Supported Projection Families

The first concrete projection family is `publish_paper_with_discussion`:

- publish a durable paper announcement to `papers`;
- create or reuse a focused forum post in `paper-discussion-forum`;
- record both OpenTofu and runtime receipts;
- copy Wendao BPMN/petgraph refs into every receipt;
- keep `paper-discussion` available for loose human chat.

The next supported projection families are:

- `bootstrap_vertical_community`: project a domain profile into Discord
  structure and seed posts;
- `agenda_session`: project a Wendao session workflow into agenda forum posts
  and scheduled events;
- `kanban_card`: project a Wendao task/work item into a Discord forum card and
  state tag.
