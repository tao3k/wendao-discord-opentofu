# User Intent Contract

The user layer should stay intent-first. Users should not need to learn the
Discord provider schema before asking for a knowledge workspace or a paper
publication.

## Primary Intents

| Intent | Example request | Durable result |
|---|---|---|
| `knowledge_operation` | Publish this paper, open a discussion, and record the result. | Discord projection envelope for Wendao workflow actions. |
| `vertical_community_profile` | Create a healthcare or business knowledge community. | Reusable profile for channels, roles, policies, and workflows. |
| `knowledge_disco` | Create a knowledge-oriented Discord space. | Category, channels, roles, permission policy. |
| `paper_announcement` | Publish this paper to `papers`. | Managed announcement or publication manifest. |
| `discussion_post` | Create discussion for this paper. | Runtime thread/post plus receipt. |
| `agenda_item` | Schedule or prepare a session. | Forum post, scheduled event, and receipt. |
| `scheduled_event` | Put an agenda session into Discord Events. | Runtime scheduled event plus receipt. |
| `kanban_card` | Track a community work item. | Forum post/card, projected state, and receipt. |
| `review_gate` | Ask reviewers to approve, revise, reject, or promote knowledge. | Private governance forum post plus receipt. |
| `space_management` | This channel is too noisy; organize it. | Proposal, HCL patch, or runtime cleanup receipt. |

## Human Review Posture

The default posture is reviewable automation:

- low-risk planning can be generated automatically;
- durable Discord state changes should be shown as a plan;
- destructive changes need explicit human approval;
- runtime discussion posts can be faster, but still produce receipts when they
  become knowledge-relevant.

Agenda and kanban are runtime work-item projections. Their forum post/card is
the Discord participation object; Wendao BPMN and petgraph keep canonical state.
The local executable examples are:

- `examples/intents/agenda_item.attention_reading_session.toml`;
- `examples/intents/scheduled_event.attention_reading_session.toml`;
- `examples/intents/kanban_card.replication_task.toml`.
- `examples/intents/review_gate.digest_promotion.toml`.

## User-Facing Shape

The durable user contract can be represented as small manifests:

```toml
kind = "paper_announcement"
target = "papers"
title = "Attention Is All You Need"
url = "https://arxiv.org/abs/1706.03762"
summary = "A transformer architecture paper."
tags = ["transformer", "attention", "paper"]
discussion = "thread_preferred"
durability = "managed"
```

The Agent may generate this from natural language and Wendao context, but the
manifest remains the reviewable bridge before projection into HCL or runtime
actions.

When a Wendao workflow step needs Discord-side work, the first reviewable object
should be a `knowledge_operation`. It groups smaller intents into one Discord
projection envelope, carries BPMN/petgraph refs from Wendao, and keeps provider
schema away from the user.

When a user wants a new vertical community, the first reviewable object should
be a `vertical_community_profile`. Wendao workflow can then choose a bootstrap
plan and produce a `knowledge_operation` that projects the profile into Discord.
