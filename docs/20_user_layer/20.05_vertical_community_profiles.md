# Vertical Community Profiles

A `vertical_community_profile` describes a reusable knowledge-community shape
for one domain. It is not provider schema and it is not Wendao workflow state.
It is a compact profile that Wendao or an Agent can use to generate a
`knowledge_operation`, HCL projection inputs, runtime seed posts, and receipts.

## Position

```text
domain need
  -> vertical_community_profile
  -> Wendao workflow chooses bootstrap plan
  -> knowledge_operation projection envelope
  -> OpenTofu structure and runtime seed posts
  -> receipts back to Wendao
```

Profiles let a user say:

> Create a healthcare evidence community.

or:

> Create a business intelligence community.

The Agent should then produce a reviewable preview instead of exposing Discord
provider fields.

## Profile Shape

```toml
kind = "vertical_community_profile"
name = "Healthcare Evidence Community"
domain = "healthcare"
review_mode = "human_approval_required"

[space]
category = "Healthcare Knowledge"
text_channels = ["announcements", "general-discussion", "knowledge"]
forum_channels = ["evidence-library", "case-review", "agenda", "kanban"]

[space.forum_tags]
agenda = ["proposed", "scheduled", "in-session", "done"]
kanban = ["todo", "doing", "review", "blocked", "done"]

[policy]
personal_advice = "not_allowed"
medical_claims = "evidence_required"
destructive_changes = "human_approval_required"
digest_publication = "review_required"

[[roles]]
key = "reader"
name = "Reader"

[[roles]]
key = "reviewer"
name = "Clinical Reviewer"

[[workflows]]
goal = "publish_evidence_with_discussion"
process_ref = "wendao://workflow/healthcare-evidence-review"
```

## Required Sections

| Section | Meaning |
|---|---|
| root | Profile identity, domain, and default review mode. |
| `space` | Semantic Discord surface to project into HCL. |
| `policy` | Domain-specific safety and governance posture. |
| `roles` | Participation roles, not necessarily Discord permission bits. |
| `workflows` | Wendao workflow families this profile expects. |

## Domain Policy

Profiles should encode domain posture without pretending to solve domain
compliance by themselves.

Healthcare profile defaults:

- no personalized medical advice;
- evidence required for medical claims;
- clinical or domain reviewer for digest publication;
- destructive Discord changes require human approval.

Business profile defaults:

- decision records require source links;
- sensitive deal or customer data should not be posted by default;
- market claims require source references;
- kanban state changes can be runtime receipts unless destructive.

## Projection Rules

- `space.text_channels` and `space.forum_channels` become HCL module inputs.
- `space.forum_tags` and `governance.forum_tags` become declared desired
  state; `sync-forum-tags` applies them because the current provider schema
  does not expose forum tag resources.
- `governance` becomes an additional private HCL category with management
  channels and role-based permission overrides.
- `roles` become HCL roles only when the profile policy allows it.
- `workflows` are copied into `knowledge_operation.workflow` refs.
- policy entries become review gates or preview warnings.
- profile name, domain, and policy must be copied into receipts.

## Preview Surface

A profile preview should be short:

```markdown
# Healthcare Evidence Community

Domain: healthcare
Review: human_approval_required

Channels:
- text: announcements, general-discussion, knowledge
- forum: evidence-library, case-review, agenda, kanban

Policy:
- personal_advice: not_allowed
- medical_claims: evidence_required
- digest_publication: review_required
```

The local renderer is:

```shell
direnv exec . just render-profile-preview
```

The profile can also be projected into OpenTofu variables for the
`vertical_community` module:

```shell
direnv exec . just render-profile-tfvars examples/intents/vertical_community.healthcare.toml
direnv exec . just plan-vertical-profile-example examples/intents/vertical_community.healthcare.toml
direnv exec . just sync-forum-tags examples/intents/vertical_community.healthcare.toml
```

That keeps the human/Agent surface compact while still producing a deterministic
HCL plan for the stable Discord structure.

## First Vertical Profiles

The first profiles should stay generic enough to reuse:

- healthcare evidence community;
- business intelligence community;
- research lab community;
- education cohort community.

Only healthcare and business examples are included now. More verticals should
be added as profile TOML, not as hard-coded provider logic.
