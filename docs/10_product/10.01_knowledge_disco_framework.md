# Knowledge Disco Framework

Knowledge Disco is the first-layer product frame for this repository. The goal
is to turn a Discord server from a mixed chat surface into a governed knowledge
workspace while preserving Discord as the live human discussion venue.

## Product Role

Knowledge Disco manages stable Discord structure:

- categories
- channels
- roles
- permissions
- durable announcements
- governance records

It does not try to make every Discord action infrastructure state. High-volume
chat, replies, transient discussion, and exploratory Agent interaction belong
to runtime tools or bots, with receipts recorded when they affect knowledge
governance.

## First-Layer Space

The initial knowledge space should expose these concepts:

| Concept | Purpose | Likely owner |
|---|---|---|
| `papers` | Durable paper announcements and pinned reading entries. | OpenTofu HCL |
| `paper-discussion` | General text discussion and follow-up chat for papers. | OpenTofu HCL plus Agent runtime |
| `paper-discussion-forum` | One focused forum post per paper discussion. | OpenTofu HCL plus Agent runtime receipt |
| `agenda` | Time-bound meetings, reading sessions, and review windows. | HCL structure, Wendao workflow runtime |
| `kanban` | Visible workflow cards and project state mirrors. | HCL structure, Wendao workflow runtime |
| `knowledge` | Curated Wendao knowledge updates. | OpenTofu HCL or Agent runtime |
| `reading-room` | Reading plans, group reading, and follow-up queues. | HCL structure, runtime content |
| `moderation-log` | Governance proposals and applied management records. | HCL structure, Agent receipts |

## Design Principle

Knowledge Disco governs space first, then carries knowledge flow. HCL owns
stable structure. Agents compile intent and context into proposals. Humans keep
review authority for risky or durable changes.

User requests that span multiple Discord actions should enter through Wendao
workflow first. This repository receives a `knowledge_operation` as the compact
Discord projection envelope, then projects, executes, verifies, and closes it
with receipts.

Vertical communities should be profile-driven. A healthcare, business,
education, or research-lab community should start from a
`vertical_community_profile` that chooses semantic spaces, roles, review
posture, and expected workflows before HCL or runtime commands are generated.
