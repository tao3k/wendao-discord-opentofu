# Management Record

This record tracks durable project decisions and operational notes.

## 2026-05-08: Knowledge Disco Direction

Decision:
Knowledge Disco is a governed Discord workspace model. HCL manages stable
Discord structure and durable announcements. Agents compile human intent and
Wendao context into reviewable projections. Runtime tools may handle dynamic
threads, posts, and replies.

Rationale:
Discord is a mixed information venue. The project needs a governance gap
between human requests, Agent action, HCL state, and live Discord behavior.

Implications:

- keep `opentofu/discord` as the HCL library root;
- keep user-facing contracts intent-first;
- represent the Human-Agent-Discord gap with policies, projections, and
  receipts;
- avoid direct dependencies on Wendao internals in this repository;
- use manifest and receipt contracts as the bridge to Wendao knowledge context.

Validation:

- `direnv exec . just check`
- `wendao audit --template johnny-decimal` is the intended documentation
  template command once `wendao` is available in this repo environment.

## 2026-05-08: Intent and Receipt Contract Skeleton

Decision:
The next durable boundary is a manifest and receipt contract, not a direct
Wendao dependency. Users and Agents can express knowledge-space, paper
announcement, discussion, and management intents as compact TOML. Execution
tools emit receipts after OpenTofu or runtime Discord work.

Rationale:
The project needs a stable gap between human language, Agent behavior, Discord
state, and Wendao context. Manifests and receipts give that gap a reviewable
shape.

Implications:

- keep examples under `examples/intents`;
- keep durable documentation under the Johnny.Decimal docs tree;
- make future CLIs consume manifests before writing HCL variables;
- make runtime discussion actions produce receipts when they affect knowledge
  governance.

## 2026-05-08: Live Discord Preflight and Apply

Decision:
The live example uses `TF_VAR_DISCORD_BOT_TOKEN` and
`TF_VAR_DISCORD_SERVER_ID` as the OpenTofu boundary variables. The helper
`direnv exec . just check-discord-auth` verifies token validity, target-server
visibility, and the guild-level permissions needed by the example before
apply.

Rationale:
The first live run proved the provider and server selection path, then failed
with Discord `403 Missing Permissions`. Encoding that as a preflight keeps the
failure actionable and avoids repeated blind apply attempts.

## 2026-05-08: Runtime Discussion Post and Forum Channel

Decision:
`paper-discussion` remains the general text discussion channel. A separate
HCL-managed `paper-discussion-forum` channel owns focused per-paper forum
posts. Discussion posts are runtime objects created from `discussion_post`
intent manifests and recorded with receipts.

Rationale:
The pinned Discord provider exposes channel and message resources, including
`discord_forum_channel`, but does not expose a thread or forum-post resource.
Creating discussion posts through a bot-side runtime action keeps the HCL state
clean while still preserving auditability.

Implementation:

- `examples/intents/discussion_post.toml` is the executable manifest;
- `opentofu/discord/modules/knowledge_space` now supports `forum_channels`;
- the live example creates `paper-discussion-forum`;
- `direnv exec . just create-discussion-post` runs
  `scripts/create_discussion_post.py`;
- the command locates the target channel, creates a public thread for text
  channels or a forum/media post for forum-like channels;
- active threads are deduped by exact title;
- runtime receipts are written under `.run/receipts`.
